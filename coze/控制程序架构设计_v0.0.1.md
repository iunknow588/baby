# 控制程序架构设计 v0.0.1

## 1. 设计目标
在不暴露 Coze 密钥的前提下，实现客户端可控、可解释、可回放的智能体与工作流控制程序。

## 2. 总体架构（RPE）
1. Router（路由）：识别用户意图并选中 `workflowId`。
2. Planner（规划）：生成可视流程步骤（用于 `flow_first` 先展示）。
3. Executor（执行）：调用 Coze 工作流或本地能力，产出结果。

对应职责：
- 客户端：Router 可视化、Planner 展示、执行进度与交互控制。
- 服务端：最终 Router 判定、Executor 真实调用、统一审计。

## 3. 组件划分
1. `workflow-registry`（服务端）
   - 维护 `workflowId -> provider(coze/local) -> version -> schema`
2. `route-engine`（前后端协同）
   - 前端快速预判；后端最终裁决并回传标准 `route`。
3. `plan-renderer`（前端）
   - 渲染 `processingFlow.steps` 与 `nextActions`。
4. `execution-gateway`（服务端）
   - 封装 Coze API、本地任务执行器、超时与重试。
5. `trace-center`（服务端）
   - 记录 `traceId`、route、workflowVersion、耗时、降级原因。

## 4. 数据协议（建议）
请求：
```json
{
  "message": "用户输入",
  "attachments": [],
  "clientRouteHint": "calligraphy_scoring"
}
```

响应（核心）：
```json
{
  "answer": "最终回复",
  "interactionMode": "flow_first",
  "processingFlow": {
    "route": "calligraphy_scoring",
    "workflowId": "wf_calligraphy_v1",
    "workflowVersion": "1.2.0",
    "steps": [
      "识别用户需求并确认评分维度",
      "执行图像与文字评分链路",
      "返回评分与改进建议"
    ],
    "nextActions": ["可继续上传下一张作业进行复评"]
  },
  "traceId": "trace_xxx"
}
```

## 5. 与现有 baby 的映射
1. 已有 `interactionMode` 可直接承载 `flow_first`。
2. 已有 `processingFlow`，补充 `workflowId/workflowVersion/traceId` 即可增强可运维性。
3. 已有 Coze 调用封装，可在 `execution-gateway` 层统一扩展本地执行器。

## 6. 安全约束
1. 浏览器不存 `COZE_API_TOKEN`。
2. 所有上游调用经后端代理并附审计信息。
3. 附件 URL 必须做签名校验与白名单校验。
