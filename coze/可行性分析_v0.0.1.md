# 可行性分析 v0.0.1（CozeLoop 参考）

## 1. 结论摘要
结论：**可行，但必须采用“前端控制 + 后端代理执行”的分层模式**。  
原因：前端可以做路由、流程可视化与交互控制；但 Coze 相关密钥、项目权限、稳定性控制与审计必须在服务端完成。

## 2. 依据与边界说明
1. 参考页面：`https://www.coze.cn/open/docs/cozeloop/what-is-cozeloop`（页面为 SPA，直接抓取多为壳页）。
2. 从 CozeLoop 前端静态资源可观察到能力域（作为辅助证据，属于反向观察）：
   - Prompt（含 optimize、version、compare）
   - Experiment
   - Dataset / Evaluation set
   - Observation / Trace
   - Evaluator / Task
3. 上述能力域说明 CozeLoop偏向“开发运维与评测平台”，并非单纯聊天 Runtime API。

## 3. 能力可行性矩阵
| 能力 | 纯客户端 | 客户端 + 后端代理 | 说明 |
|---|---|---|---|
| 会话 UI、流程展示、步骤确认 | 可行 | 可行 | 前端天然能力 |
| 路由判定（规则/轻量模型） | 有条件可行 | 可行 | 涉及模型与策略更新时建议后端托管 |
| 工作流编排定义读取 | 有条件可行 | 可行 | 只读配置可前端缓存，发布/变更建议后端 |
| 调用 Coze 聊天/工作流执行 | 不建议 | 可行 | 需要服务端持有 Token 与权限 |
| Prompt/实验/评测任务管理 | 不建议 | 可行 | 多为平台权限操作，应集中在后端 |
| 观测追踪、日志聚合、审计 | 不可行 | 可行 | 需服务端统一落盘、脱敏与告警 |
| 附件安全处理（URL签名/校验） | 不可行 | 可行 | 涉及存储与安全策略 |

## 4. 对当前 baby 项目的适配性
当前项目已有基础，适配成本低：
1. 已有 Coze 调用封装与环境变量校验（`COZE_*`）。
2. 已有 `processingFlow` 输出，可作为控制程序的标准执行回执。
3. 已有 `interactionMode`（`direct | flow_first`），可直接支持“先流程后执行”。

## 5. 推荐方案
1. 前端（控制层）：负责路由解释、流程可视化、用户确认、状态回放。
2. 后端（执行层）：负责 Coze/本地能力编排、鉴权、重试、审计、降级。
3. 配置层：工作流注册中心（版本化），支持 Coze 工作流与本地任务混排。

## 6. 主要风险与规避
1. 风险：浏览器泄露凭证。  
   规避：所有 Coze 私密凭证只放服务端；前端只拿短时会话令牌。
2. 风险：流程定义与代码漂移。  
   规避：工作流配置版本化 + 启动时一致性检查。
3. 风险：出现“看起来执行了，实际没命中正确工作流”。  
   规避：每次响应强制返回 `route/workflowId/version/traceId`。
